# Database — switching to Postgres (Drizzle)

The app ships with two interchangeable backends behind one async interface (`src/server/data-store.ts`):

- **InMemoryStore** (`src/server/store.ts`) — the default. Re-seeds on every boot. Great for the demo; state is per-process and not durable.
- **DrizzleStore** (`src/server/store-drizzle.ts`) — **Postgres**. Used automatically when `DATABASE_URL` is set.

`getStore()` picks the backend: `DATABASE_URL` present → Postgres, else in-memory. **No code changes are needed to switch — it's one env var.**

## Go to Postgres in 4 steps

```bash
# 1. Point at a Postgres database (Neon, RDS, local docker, …)
export DATABASE_URL="postgres://user:pass@host:5432/aithreus"

# 2. Create the tables from the Drizzle schema
npm run db:push          # or: npm run db:generate && apply drizzle/*.sql via your migration tool

# 3. Seed demo data (same data the in-memory store uses)
npm run db:seed

# 4. Run — the app now reads/writes Postgres
npm run dev
```

## Files
- **Schema:** `src/db/schema.ts` — every table, typed to the domain (string-literal unions via `.$type<>()`). Date/time columns are ISO `text` to match the app's string types.
- **Migrations:** `drizzle/` (generated by `npm run db:generate`). `drizzle.config.ts` holds the config.
- **Connection:** `createDrizzleStore()` in `src/server/store-drizzle.ts` builds a `pg` Pool from `DATABASE_URL`.
- **Seed:** `scripts/seed.ts` (`npm run db:seed`) calls `store.reset()`, which wipes + inserts the seed and pre-publishes live sites.

## Verify without a server
`npm run db:verify` runs the **real DrizzleStore** against embedded Postgres (PGlite) and asserts visibility filtering, seed-time publish, mutations, and analytics. Use it in CI to guard the data layer.

## Notes for production
- **Migrations:** prefer `db:generate` + a migration runner (`drizzle-kit migrate`) over `db:push` in prod.
- **Pooling:** for serverless (Amplify/Lambda), use a pooled connection string (e.g. RDS Proxy / PgBouncer / Neon pooled endpoint) to avoid connection exhaustion.
- **FKs/indexes:** the prototype schema declares no foreign keys or extra indexes (kept simple). Add FKs and indexes (`config_id`, `site_id`, `event.ts`) before scale.
- **Analytics volume:** the `events` table grows fast. See INTEGRATION.md §Analytics for the ClickHouse seam; at minimum add time/site indexes and a retention/rollup job.
- **`reset()` is destructive** (used by seed + the dev reset endpoint). Never expose it in production.
